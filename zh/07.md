第 7 章


Context Engineering：不是塞得越多越好，而是刚刚好

如果说大模型是 Agent 的"大脑"，那上下文就是 Agent 的"视野"。大脑再聪明，看不见东西也白搭。但你知道吗？视野不是越大越好——挡风玻璃太大，车会漏风；视野里东西太多，司机会走神。上下文工程，就是给 Agent 装一扇刚刚好的窗户。

"爽！"小明伸了个懒腰，顺手点开了 AI 服务商的账单页面，想看看这个月花了多少钱。

这一看不要紧，小明差点从椅子上摔下来。

小明

八千多？！这怎么可能！我明明只用了不到一个月啊！平均一天快三百块？我这是在烧钱还是在炼丹？

小明

对啊！不是说上下文越全越好吗？我把所有文件都给它了，让它全面了解项目，这样写出来的代码才不会错啊！

挂了电话，小明半信半疑。他把 Agent 的上下文配置改了改——不再加载所有文件，只加载当前需要修改的几个相关文件。

"这……这到底是为什么？"小明彻底懵了，"给的信息少了，效果反而更好了？"

老王笑了笑，约小明周末到公司旁边的咖啡馆，说要好好给他上一课——**上下文工程**。

## 二、Context 是个"双向放大器"

周六下午，阳光正好。老王端着咖啡，慢条斯理地给小明讲起了上下文工程的第一个核心概念。

### 好的放大好的，坏的放大坏的

"小明，你知道为什么信息少了，效果反而更好吗？"老王放下咖啡杯，"因为 Context 本质上是一个**双向放大器**。"

"双向放大器？"小明挠挠头，"什么意思？"

Context 是双向放大器——好的输入被放大得更好，坏的输入被放大得更烂

老王举了个例子：

想象你有一个音响。你输入一首音质很好的歌，开了放大器，出来的声音更加震撼动听。但如果你输入的是一首充满杂音的歌，开了放大器呢？出来的不是更好听的音乐，而是更刺耳的噪音。

Context 就是这个放大器。

Context 是双向放大器：好的工程实践会被放大，烂的代码结构也会被放大。

老王给小明讲了他亲身经历的两个项目：

**第一个项目**，代码规范做得极好——命名清晰、注释完整、架构分层明确、每个函数职责单一。团队引入 AI 编码助手后，效率直接翻倍。AI 不仅能照着现有风格写代码，甚至还能主动发现一些潜在的 bug，把规范执行得比人还彻底。

**第二个项目**，祖传屎山——变量名叫 `a`、`b`、`c`，一个函数两千行，到处都是复制粘贴的重复代码，注释里写的是"这里不要改，改了会崩"。结果呢？AI 一上手，写出来的代码比原来的还烂。它学到了所有坏毛病，而且因为"学"得太认真，把那些烂模式复制得更加淋漓尽致。

小明

我去……这么恐怖？那屎山项目岂不是没救了？

老王

也不是没救，但你不能直接把整座屎山都扔给 AI。你得先清理——把相关的、干净的、正确的信息挑出来给它。这就是上下文工程干的事：**给 AI 看好的东西，它就会产出好的结果；给 AI 看烂的东西，它就会产出更烂的结果。**

### 代码质量差？AI 会照着烂模式写得更烂

老王给小明看了一个真实的例子。

有个团队的代码里，错误处理写得非常随意——基本都是 `try { ... } catch (e) { console.log(e) }`，捕获异常后就打个日志，啥也不干。他们把整个代码库塞给 AI 让它加新功能，结果 AI 写出来的新代码，错误处理也是一模一样的套路，甚至连日志的格式都不带变的。

"这就叫'近朱者赤，近墨者黑'，"老王笑着说，"AI 没有判断力，它只会模仿上下文里的模式。你给它看十段烂代码，它就会写出第十一段烂代码。"

⚠️ 注意

**垃圾进，垃圾出（Garbage In, Garbage Out）**——这句人工智能领域的老话，在 Context Engineering 时代依然成立。而且因为大模型的"模仿能力"太强，垃圾进去了，出来的可能是更多、更像的垃圾。

### 工程规范好？AI 会把规范执行得比人还彻底

反过来，如果你的上下文里都是高质量的代码和清晰的规范，AI 的表现会让你惊喜。

老王说了另一个例子：有个团队花了一周时间，把代码规范、设计原则、最佳实践整理成了一份清晰的文档，放在 Agent 的系统提示词里。结果发现，AI 写出来的代码不仅完全符合规范，而且比很多老员工写的还"标准"——它不会因为偷懒而跳过错误处理，不会因为图方便而破坏分层，不会因为忘记而漏掉注释。

老王

人写代码，总有状态不好的时候，总有偷懒的时候，总有忘记规范的时候。但 AI 不会——只要上下文里的规范足够清晰，它每次都会严格执行。这就是 Context 作为放大器的正面力量。

小明

所以说……上下文工程的第一步，就是确保我们给 AI 看的东西本身是高质量的？

老王

没错！这也是为什么很多团队在引入 AI 之前，先做一轮代码清理和规范建设。不是为了人，而是为了让 AI 有一个好的"学习环境"。**你的代码库质量，决定了 AI 输出的上限。**

## 三、上下文的"预算管理"

讲完了双向放大器的概念，老王话锋一转，回到了小明最关心的问题——钱。

### Token 就是钱：每一轮对话都在烧钱

"小明，你知道你那八千多块钱是怎么花的吗？"老王问道。

小明摇摇头："我就知道每次对话消耗很多 Token，但具体怎么算的，我还真没仔细研究过……"

上下文油表——Token 就是钱，每一轮对话都在烧钱

老王给小明算了一笔账：

- 大模型的计费方式，是按 Token 算钱的，输入和输出都要算
- 什么是 Token？大概就是一个字到一个半字左右（中文），或者四分之三个英文单词
- 每一轮对话，你都要把**整个上下文历史**重新发给模型——不是只发最新的一句话
- 所以上下文越长，每一轮对话的成本就越高
- 而且这是"叠加"的——第10轮对话，你已经把前9轮的内容付了10次钱

小明瞪大了眼睛："等等！你是说，我第10轮说的那句话，其实我已经为前面9轮的内容付了10次钱？"

老王点点头："没错。这就像你每次打车，司机都要从起点重新开一遍——你每说一句话，都要把之前所有的话重新'读'一遍。所以上下文越长，边际成本越高。"

💡 小明算算账

假设你的上下文有 5 万 Token，每一轮对话输入成本大约是 0.5 元，输出 0.5 元。一天聊 50 轮，就是 50 元。一个月就是 1500 元。但如果你把上下文控制在 1 万 Token 以内，成本直接降到五分之一——一个月只要 300 元。

### 上下文窗口不是越大越好：塞太满效果反而下降

"钱的问题还好说，大不了多花点钱。"老王话锋一转，"但更关键的问题是——**上下文塞太满，效果反而会下降。**"

小明又懵了："啊？窗口大不是好事吗？模型厂商都在比谁的窗口大啊！128K、200K、1M……越大越厉害啊！"

老王笑了笑，问了一个问题："小明，你考试的时候，是开卷考高分，还是闭卷考高分？"

"当然开卷啊！"小明不假思索。

"那如果开卷考试，给你一整个图书馆呢？"老王继续问，"让你在一百万本书里找答案，你觉得你能考得更好吗？"

小明愣住了。

老王

道理是一样的。上下文窗口大，不代表 AI 能有效利用。研究表明，当上下文过长时，模型容易出现**"中间遗忘"**——就是开头和结尾的信息记得比较清楚，中间的信息容易被忽略。

小明

这不就跟人一样嘛……上课只记得老师开头说什么和结尾布置了什么作业，中间讲啥来着？

老王

哈哈，对！所以上下文工程的第二个核心原则就是——**窗口大是能力，但能不能用好是水平**。就像车的最高时速能跑 200 码，但不代表你在城市道路上就得开 200 码。

### 小明的"上下文油表"：实时监控还剩多少空间

老王建议小明给自己的 Agent 装一个"上下文油表"——实时显示当前上下文已经用了多少 Token，还剩多少空间。

"这就跟开车看油表一样，"老王说，"你不能等油烧干了才发现没油了。你得时刻关注还剩多少油，提前规划。"

老王给小明列了几个关键的"油表刻度"：

1

安全区：0% \~ 50%

上下文还很充裕，AI 注意力集中，输出质量稳定。可以放心地加内容。

2

警戒区：50% \~ 70%

开始注意了。需要考虑是否可以压缩一些不重要的信息，或者把老的对话总结一下。

3

危险区：70% \~ 90%

必须采取行动了！压缩、总结、清理——把不重要的信息移出去，保证关键信息不丢失。

4

爆表区：90% 以上

随时可能"翻车"。AI 可能开始遗忘关键信息，输出质量明显下降。必须立即清理。

### 128K、200K、1M…… 窗口越大越好吗？

聊到这里，小明忍不住问出了那个藏在心里很久的问题：

小明

老王，那你说现在厂商都在拼上下文窗口，从 32K 到 128K，到 200K，再到 1M，甚至有的说能到 10M……这玩意儿到底有没有用？

老王

有用，但不是你想的那种"有用"。大窗口的价值不在于"把所有东西都塞进去"，而在于**"不用频繁地倒腾东西"**。就像汽车油箱大——不是让你一次把油加满就永远不加了，而是让你不用频繁跑加油站。

老王解释说，大上下文窗口有两个主要价值：

**第一，减少"搬运成本"。**如果窗口很小，你就得频繁地把旧信息搬出去、把新信息搬进来，这个"搬运"过程本身就需要消耗计算资源和精力。窗口大一些，你就有更多的"缓冲空间"。

**第二，处理长文档场景。**有些任务天生就是长的——比如读一整本小说、分析一份几十页的合同、理解一个大型代码库的架构。这种时候，大窗口就是刚需。但即使是这些场景，也需要上下文工程——不是整本书囫囵吞枣地塞进去，而是有策略地组织和检索。

✅ 老王总结

大窗口是"上限能力"，上下文工程是"下限保障"。窗口再大，不会管理也白搭；窗口不大，管好了照样能用好。**就像开车——车能跑多快是车的事，你能开多稳是你的事。**

### 顺序很重要：开头和结尾记得最牢

"小明，你有没有发现，"老王说，"你看一篇文章，开头和结尾总是记得最清楚，中间讲了啥反而容易忘？"

小明点点头："对啊！我背单词也是，abandon 记得最牢，最后几个也有印象，中间的……"

"AI 也一样，"老王说，"这在心理学上叫'系列位置效应'（Serial Position Effect）——开头的叫'首因效应'，结尾的叫'近因效应'。大模型也有这个特点。"

小明

那这意味着……我们应该把最重要的信息放在最前面和最后面？

老王

正解！上下文的排列顺序不是随便排的。**系统提示词放最前面（首因效应），当前任务放最后面（近因效应），中间放辅助信息。**这就是为什么 Agent 的架构里，系统指令永远在最顶部，用户的最新问题永远在最底部。

老王给小明画了一个简单的上下层结构图：

🎯

系统指令（System Prompt）

角色定义、核心规则、输出格式 —— 放最前面，确保 AI 第一时间看到

📚

背景知识（Context）

相关文档、代码片段、历史对话 —— 放中间，作为参考资料

❓

当前任务（User Query）

用户最新的问题或指令 —— 放最后面，AI 刚"看完"就回答

### 结构化：用 XML 标签、Markdown 标题分区

"光有顺序还不够，"老王继续说，"你还得让 AI 知道——哪部分是什么内容。这就叫**结构化**。"

老王举了个例子。如果你把一堆信息乱七八糟地堆在一起，AI 可能分不清哪些是指令、哪些是例子、哪些是数据。但如果你用清晰的结构分开，AI 的理解准确率会大幅提升。

常见的结构化方法有：

- **XML 标签分区：**用 `<system>`、`<context>`、`<query>` 等标签把不同部分包起来
- **Markdown 标题：**用 #、##、### 给内容分层次，一目了然
- **代码块包裹：**代码用 \`\`\` 包起来，和文字区分开
- **明确的分隔线：**用 --- 或 === 把不同的信息块分开

💡 小技巧

不要小看这些"格式"。对人来说，格式只是好看；对 AI 来说，格式是**理解的路标**。清晰的结构能帮助 AI 快速定位关键信息，减少"读错行"的概率。

### 层级化：重要信息放前面，细节放后面

"什么叫层级化？"小明问。

"就像写邮件，"老王说，"主题要明确，重点放开头，详情放后面。"

老王解释说，好的上下文组织应该是"金字塔结构"——最核心的信息在最上面，然后逐层展开细节。AI 即使只看了前面一部分，也能抓住核心意思。如果它需要更多细节，再往下看。

比如你要让 AI 修改一个函数，上下文的组织应该是：

1. **第一层（核心）：**你要做什么？——"请修改用户登录函数，增加手机号登录方式"
2. **第二层（背景）：**为什么要做？——"产品需求文档见附件，目标是支持手机号+验证码登录"
3. **第三层（参考）：**相关代码在哪？——"当前登录函数在 auth.js 第 23-56 行，验证码服务已在 sms.js 中实现"
4. **第四层（细节）：**具体要求是什么？——"需要兼容原有账号密码登录，错误码规范参考第 4.2 节……"

老王

对！这就是上下文工程的艺术——**不是塞得越多越好，而是刚刚好。该看的都看到，不该看的不凑热闹。**而且就算看不完，至少也看了最重要的。

### Skill 的三大件：描述 + 文档 + 脚本

🏷️

描述（Description）

🧩

独立维护

每个 Skill 可以独立开发、独立测试、独立更新，互不影响

🔄

一个团队写的 Skill，另一个团队可以直接用，不用重复造轮子

📦

不同的任务场景，可以加载不同的 Skill 组合，灵活搭配

📈

持续进化

Skill 可以不断迭代优化，整个 Agent 的能力也随之水涨船高

故事

老王说他认识的一个团队，一开始把所有的代码规范、审查标准、部署流程都写在一个巨大的系统提示词里，有好几千个 Token。每次改一个规范都得小心翼翼，生怕改坏了别的。

### MCP 与上下文工程的关系

"那 MCP 和上下文工程是什么关系呢？"小明问，"MCP 不是提供了更多信息吗？那上下文不会爆炸吗？"

老王赞许地点点头："好问题！这就是上下文工程的精髓——**有能力获取更多信息，不等于要把所有信息都塞进来。**"

他解释道，MCP 提供的是"获取信息的通道"，但具体拿什么信息、拿多少、什么时候拿，这些都是上下文工程要管的事。

就像汽车有 GPS 导航，但导航不会把整个中国的地图都显示在屏幕上——它只会显示你当前位置附近的路，以及你前方的路线。MCP 是那个"导航系统"，上下文工程是那个"地图缩放和过滤"。

✅ 核心洞察

MCP 解决的是"能不能看到"的问题，Context Engineering 解决的是"该看什么"的问题。**一个是能力，一个是策略；一个是拓展边界，一个是优化效率。**两者结合，Agent 才能既看得广，又看得准。

### 上下文压缩：太长了就先总结

"第一个技巧，**上下文压缩**，"老王说，"简单来说就是——太长的内容，先让 AI 自己总结一下，再把总结放进上下文里。"

小明眼睛一亮："哦！这招我喜欢！就像看电影解说一样，两个小时的电影，五分钟给你讲完。"

"对，就是这个意思，"老王笑着说，"比如有一份 50 页的需求文档，你不需要全部塞进上下文。你可以先让 AI 读一遍，总结出'核心需求 5 条、关键约束 3 条、待确认问题 2 条'，然后把这个总结放进上下文。Token 消耗可能只有原来的十分之一，但关键信息一个没少。"

老王补充说，上下文压缩有几种常见的做法：

- **摘要压缩：**把长文档总结成短摘要，保留核心意思
- **提取关键信息：**只提取人名、地名、时间、数字等关键实体
- **分级摘要：**先总结每一段，再总结每一章，最后总结全文
- **对话压缩：**把前面的对话历史总结成一段"已讨论内容"

### 动态检索：需要的时候再找资料

"第二个技巧，**动态检索**，也就是大家常说的 RAG（Retrieval-Augmented Generation），"老王说，"核心思想就是——不把所有资料都带在身上，需要的时候再去图书馆查。"

小明点点头："这个我知道！就是向量数据库对吧？"

"对，但不仅仅是向量数据库，"老王说，"动态检索的本质是**'按需取数'**——不管是用向量搜索、关键词搜索、还是数据库查询，只要是'需要的时候才把相关信息拿进来'，都叫动态检索。"

老王给小明举了个例子：

你有一个知识库，里面有 1000 篇技术文档。你不需要把 1000 篇都塞进上下文。当用户问一个问题时，你先去知识库搜索最相关的 3-5 篇，只把这几篇放进上下文里。这样既省 Token，又能保证 AI 看到的都是相关的内容。

💡 老王的比喻

动态检索就像汽车的**自动大灯**——天黑了自动亮，天亮了自动关。不需要一直开着，但需要的时候它一定在。上下文工程就是那个"光感传感器"——判断什么时候需要补充信息、需要补充什么信息。

### 上下文缓存：重复的内容不重复算钱

"第三个技巧，**上下文缓存**，"老王说，"这个是省钱神器。"

"缓存？"小明疑惑，"怎么个缓存法？"

老王解释道："你想啊，每一轮对话，你都要把整个上下文重新发给模型。但这里面很多内容其实是不变的——比如系统提示词、项目背景文档、已加载的 Skills……这些内容每一轮都一样，但你每一轮都在为它们付钱。"

小明瞪大了眼睛："对啊！这不是重复收费吗？"

"所以现在很多大模型厂商推出了'上下文缓存'功能，"老王说，"如果你的上下文里，前面一大段和上一轮是一样的，那这部分就只收一次钱，或者打个大折扣。这样能省不少钱。"

小明

那怎么才能最大化利用缓存呢？

老王

很简单——**把不变的内容放前面，把变化的内容放后面。**系统提示词、项目规范这些基本不变的，放最前面。对话历史、当前任务这些每轮都变的，放后面。这样前面不变的部分就能被缓存住，不用重复计费。

### 上下文审计：AI 到底看了什么？

"最后一个技巧，**上下文审计**，"老王的语气变得严肃起来，"这个很多人忽略，但非常重要。"

"上下文审计？审什么？"

"审计 AI 到底看了什么，"老王说，"你有没有过这种经历——AI 回答错了，你去查原因，结果发现是它'看错了'某个信息？或者更严重的——它看到了不该看的信息？"

小明想了想，还真有。上次他让 AI 写一段代码，结果 AI 用了一个早就废弃的 API。查了半天，发现是上下文里有一份旧版文档，AI 照着旧版写了。

老王

上下文审计就是——**每一次 AI 回答完，你都要能追溯：它这次回答到底看了哪些信息？这些信息对不对？全不全？有没有过期？**这不仅是质量保障，也是安全保障。

小明
