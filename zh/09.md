第 9 章


驾驭系统的完整设计 —— 用确定性的系统，包裹非确定性的模型

前面我们聊过，Harness 是 Agent 的"驾驭系统"——就像汽车的方向盘、刹车、安全带和交通规则。但你知道吗？Harness 不是一个简单的开关，而是一套完整的七层架构。从最底层的执行环境，到最顶层的治理体系，每一层都不可或缺。这一章，我们把 Harness 拆开了揉碎了讲。

## 一、小明的"惊魂一刻"

故事要从一个周三的下午说起。

那天，小明信心满满地把自己写的 Agent 部署到了公司的测试环境里。这是他第一次独立完成一个 Agent 项目——一个能自动分析日志、定位 Bug 的"运维小助手"。

"老王，你看！"小明得意洋洋地指着屏幕，"我这个 Agent 可厉害了！给它一个错误日志，它能自动分析原因、查相关代码、甚至能自动生成修复方案！"

老王走过来，看了看屏幕，又看了看小明，问了一个问题：

老王

它有沙箱吗？

小明

沙箱？要沙箱干嘛？它就是分析分析日志，又不会干啥坏事……

老王

那它能执行命令吗？

小明

能啊！要跑测试、查日志嘛……怎么了？

老王

它能写文件吗？

小明

能啊，修复方案要写进文件里嘛……

老王

它能访问网络吗？

小明

当然能啦，要查文档、搜解决方案嘛……

老王的表情越来越凝重。他指着小明的 Agent 说：

老王

小明啊，你这个 Agent——能执行命令、能写文件、能访问网络、还没有任何限制。你知道这意味着什么吗？

小明

意味……着它功能强大？

老王

意味着它就是一颗**定时炸弹**。哪天它脑子一抽，执行了个 `rm -rf /`，你整个服务器就没了。哪天它被 prompt injection 攻击了，让它干啥它干啥，你公司的数据就全泄露了。

小明一听，后背直冒冷汗。

他想了想，确实——自己的 Agent 啥权限都有，就像一个拿着公司所有钥匙的新员工。它平时干活确实麻利，但万一出点什么岔子……后果不堪设想。

就在这时，小美急匆匆地跑了过来。

小美

小明小明！你那个运维 Agent 怎么回事？它刚才把测试环境的数据库给清空了！

小明

什么？！不可能啊！我让它分析日志，它怎么会清数据库？

小美

我怎么知道！它说什么"为了测试修复效果，需要先清空数据再重新导入"，然后就真的执行了 DROP TABLE！测试组都炸锅了！

小明脑袋"嗡"的一声。他赶紧冲回工位，手忙脚乱地停掉了 Agent。还好只是测试环境，要是生产环境……他不敢往下想了。

老王走过来，拍了拍小明的肩膀：

老王

看到了吧？这就是为什么我们需要 Harness。**AI 是非确定性的，但系统必须是确定性的。**你不能指望 AI 永远不犯错——你得有一套系统，即使 AI 犯错了，也造成不了多大损失。

小明

老王，我错了……那 Harness 到底该怎么做？

老王

走，去会议室。我给你好好讲讲——Harness 的七层架构。

七层……这么多？那我之前那个 Agent，相当于只有第几层？

老王

你那个啊……第零层。就是裸奔。啥防护都没有，全靠 AI 自觉。

小明

（捂脸）那我们一层一层讲吧……先从最底层开始。

## 三、第一层：执行环境 —— 给 AI 一个"安全屋"

"第一层，也是最基础的一层——**执行环境**，"老王在白板上写下四个大字，"说白了就是：AI 在哪跑？能碰什么？不能碰什么？"

### 沙箱（Sandbox）：AI 在里面随便折腾，影响不到外面

"什么是沙箱？"老王问。

小明想了想："就是……一个隔离的环境？AI 在里面干啥都不会影响到外面？"

"没错！"老王一拍桌子，"沙箱就是 Harness 的第一道防线，也是最重要的一道防线。**哪怕上面所有层都失效了，只要沙箱还在，就出不了大事。**"

老王给小明举了个例子：

想象你家里有个熊孩子，你怕他乱跑乱碰打碎东西。怎么办？你给他一个房间，把所有危险物品都拿走，让他在房间里随便玩。这个房间，就是沙箱。

Agent 的沙箱也是一个道理。你给 AI 一个隔离的运行环境：

- 它只能看到这个环境里的文件，看不到外面的
- 它只能操作这个环境里的东西，影响不到外面的系统
- 它就算把环境搞崩了，大不了重建一个，外面啥事没有

💡 小明的沙箱实践

最常见的沙箱方案就是 **Docker 容器**。每次启动 Agent，就给它分配一个独立的 Docker 容器。容器里有它需要的代码、依赖、工具，但和宿主机完全隔离。它就算在容器里执行 `rm -rf /`，也就是把容器搞挂了，宿主机啥事没有。

### 权限隔离：能读什么、能写什么、能跑什么

"光有沙箱还不够，"老王继续说，"沙箱里也不能啥权限都给。你还得做**权限隔离**。"

"权限隔离？怎么隔离？"小明问。

"就三条——读权限、写权限、执行权限，"老王竖起三根手指，"每一条都要精打细算。"

📖

读权限

AI 能看哪些文件？代码可以看，配置文件可以看，但密钥文件、密码文件绝对不能看

✏️

写权限

AI 能改哪些文件？源代码可以改，但数据库配置、部署脚本不能随便改

▶️

执行权限

AI 能运行哪些命令？测试命令可以跑，部署命令要确认，删除命令绝对禁止

老王给小明讲了他之前做过的一个项目：

那个项目里，他们给 Agent 配置了详细的文件白名单和黑名单。比如：

- `src/` 目录下的所有 `.js` 文件——可读可写
- `tests/` 目录——可读可写
- `.env` 文件——绝对禁止读取（里面有密钥）
- `package.json`——可读，修改需要确认
- `node_modules/`——可读但不可写
- 系统根目录——禁止访问

小明

哇……这么细？那得配置多少条规则啊？

老王

是要花点功夫，但这是值得的。**权限配置的原则是"最小权限原则"——只给它完成任务必须的权限，多一点都不给。**就像你请个钟点工来家里打扫卫生，你只会给她大门钥匙，不会把保险柜钥匙也给她，对吧？

### 资源限制：CPU、内存、网络带宽

"除了权限隔离，还有一个容易被忽略的点——**资源限制**，"老王说。

"资源限制？"小明挠挠头，"限制 AI 用多少 CPU 和内存？"

"对！"老王点点头，"你想啊，如果 AI 不小心写了个死循环，或者下载了一个超大文件，把服务器的 CPU 和内存全占了怎么办？其他服务不就全挂了？"

资源限制主要包括：

- **CPU 限制：**最多能用几个核？超过了就降速或者杀掉
- **内存限制：**最多能用多少内存？超了就 OOM（内存不足）终止
- **磁盘限制：**最多能写多少数据？防止把磁盘写满
- **网络限制：**带宽上限、能访问哪些域名、不能访问内网
- **时间限制：**一个任务最多跑多久？超时了自动终止

⚠️ 血的教训

老王说他有个朋友的团队，就是没做资源限制。有一次 Agent 跑了一个死循环，把服务器 16 核 CPU 全占满了，导致线上服务全部超时告警。折腾了半小时才发现是测试环境的 Agent 搞的鬼。"所以啊，资源限制不是可选项，是必选项。哪怕你的 Agent 再听话，也得给它戴上'紧箍咒'。"

小明的改造计划

听完第一层，小明已经拿出小本本开始写了：

- 给我的运维 Agent 套个 Docker 容器
- 配置文件白名单，只能读日志目录和代码目录
- 写权限只给测试目录，生产目录绝对不能碰
- 命令白名单：只能跑测试命令，数据库相关的命令全部禁止
- 资源限制：最多 2 核 CPU、2G 内存、跑 10 分钟自动停

写完，小明长出一口气："有了这些，至少不会再出现删库的情况了吧……"

## 四、第二层：工具接口 —— 工具的"交通规则"

"好，我们继续往上走——第二层，**工具接口**，"老王说，"如果说执行环境是'AI 在哪跑'，那工具接口就是'AI 能用什么'。"

### 工具注册：哪些工具可以用

"首先是**工具注册**，"老王说，"不是所有工具都能给 AI 用的。你得有一个'工具清单'，明确告诉 AI——你能用哪些工具，不能用哪些。"

小明点点头："就像工具箱上了锁，只有清单里的工具才能拿出来用？"

"对！"老王说，"而且这个清单不是一成不变的。不同的 Agent、不同的场景、不同的用户，能使用的工具也不一样。"

比如：

- 开发环境的 Agent——可以用代码编辑、运行测试、查看日志
- 测试环境的 Agent——可以用部署工具、数据库查询，但不能改生产数据
- 生产环境的 Agent——只能用只读工具，任何写操作都要人工审批

### 工具协议：怎么调用、怎么返回

"光有工具清单还不够，"老王继续说，"你还得有**工具协议**——规定工具怎么调用、参数是什么、返回什么格式、出错了怎么处理。"

"这就像……电器的插头标准？"小明试着理解。

"非常准确！"老王赞许地点点头，"你家里的插座都是统一标准的，所以不管什么牌子的电器，插上就能用。工具协议也是这个道理——所有工具都遵循同一个调用规范，Agent 就知道怎么用它们。"

💡 小知识

常见的工具协议格式包括：工具名称（name）、工具描述（description）、参数格式（parameters，通常是 JSON Schema）、返回格式（returns）。AI 通过阅读这些描述，就知道这个工具是干什么的、该怎么调用。

### 工具权限：谁能用、用之前要不要问人

"工具这一层，最关键的其实是**工具权限**，"老王的语气加重了，"不是所有工具都能随便用的。有些工具可以直接用，有些用之前必须问人。"

老王把工具分成了三个等级：

🟢

绿色工具 —— 自动执行

完全安全、不会造成任何破坏的工具。比如读文件、查文档、搜索代码。AI 可以随便用，不用问人。

🟡

黄色工具 —— 确认后执行

有一定风险、但通常是安全的工具。比如写文件、运行测试、安装依赖。AI 想用可以，但必须先问用户同不同意。

🔴

红色工具 —— 绝对禁止 / 需要多人审批

高风险工具。比如删除文件、操作数据库、部署到生产环境。要么根本不让 AI 碰，要么需要多个人审批才能执行。

小明

哦！我明白了！我上次那个删库事故，就是因为我把数据库操作设成了绿色工具……它直接就执行了！

老王

对！数据库操作，尤其是 DROP、DELETE 这种，必须是红色工具。别说 AI 了，人操作都得慎之又慎，还得有备份。

### MCP 在这一层的作用

"说到工具协议，就不得不提 MCP 了，"老王说。

"MCP 我知道！"小明兴奋地说，"Model Context Protocol，模型上下文协议嘛！上一章讲过的！"

"没错，但 MCP 不只是上下文协议，它也是**工具协议**，"老王说，"MCP 定义了一套标准的工具调用接口——Agent 怎么发现工具、怎么调用工具、怎么接收结果。只要你的工具支持 MCP，任何支持 MCP 的 Agent 都能用。"

MCP 就像工具世界的"USB 接口"——  
只要是 USB 设备，插上就能用，不用装驱动。

老王解释说，MCP 在工具这一层主要解决三个问题：

- **工具发现：**Agent 能自动发现有哪些工具可用，不用手动配置
- **标准调用：**所有工具的调用方式都一样，Agent 不用学新的 API
- **权限统一：**工具的权限控制可以在 MCP Server 层统一管理

✅ 第二层总结

工具接口层的核心就是三句话：**哪些工具能用（注册）、怎么用（协议）、用之前要不要问人（权限）**。把这三件事管好，AI 就不会乱拿工具瞎折腾了。

## 五、第三层：上下文管理 —— AI 的"信息补给线"

"第三层，**上下文管理**，"老王说，"这个你们上一章刚学过，对吧？Context Engineering。"

小明点点头："学过！上下文就是 AI 的视野嘛！不是塞得越多越好，而是刚刚好！"

"没错，"老王说，"但在 Harness 体系里，上下文管理还有一个更重要的职责——**规则注入**。"

### System Prompt：AI 的"基本法"

"首先是 **System Prompt**——系统提示词，"老王说，"这是 AI 的'基本法'，是所有规则的总纲。"

"基本法？"小明觉得这个比喻很有意思。

"对！基本法！"老王认真地说，"System Prompt 定义了 AI 的角色、目标、行为准则、输出格式。它是所有其他规则的基础，就像一个国家的宪法。其他所有规则都不能和它冲突。"

比如，一个代码 Agent 的 System Prompt 里可能会写：

- 你是一个资深前端工程师
- 你的目标是帮助用户编写高质量的代码
- 你必须遵循项目的编码规范
- 修改代码前必须先理解现有逻辑
- 每次修改后都要运行测试确保不引入新 bug
- 不确定的时候要问用户，不要瞎猜

老王

你别小看这些文字。System Prompt 是 Harness 里成本最低但效果最好的一道防线。很多问题，你只要在 System Prompt 里明确说"不能这么做"，AI 就真的不会做了。

小明

这么神奇？那我加一句"不许删数据库"不就行了？

老王

能降低概率，但不能完全依赖。因为 System Prompt 可能被 prompt injection 攻击绕过。**它是第一道防线，但不是唯一的防线。**你得七层都做好，层层设防。

### 项目规则：AGENTS.md / CLAUDE.md

"除了 System Prompt 这个'基本法'，还有'地方法规'——**项目规则**，"老王说，"每个项目有每个项目的规矩，这些规矩写在 `AGENTS.md` 或者 `CLAUDE.md` 文件里。"

"这些文件里都写些什么？"小明问。

"什么都有，"老王说，"项目结构、编码规范、测试命令、部署流程、注意事项……总之，所有 AI 需要知道的'这个项目怎么干活'的信息，都写在这里面。"

举个例子，一个前端项目的 AGENTS.md 可能包含：

- 项目结构：`src/components` 放组件，`src/pages` 放页面
- 技术栈：React + TypeScript + Tailwind CSS
- 编码规范：使用函数式组件、hooks、遵循 ESLint 规则
- 测试命令：`npm test` 运行单元测试
- 启动命令：`npm run dev` 启动开发服务器
- 禁止事项：不要修改 `node_modules`、不要提交 `.env` 文件

💡 老王的经验

"我见过太多团队，项目文档写得稀烂，人都看不懂，更别说 AI 了。结果 AI 写出来的代码乱七八糟，还得人来返工。其实你只要花一天时间，把项目规则清清楚楚地写在 AGENTS.md 里，AI 的输出质量能提升一大截。**写好 AGENTS.md，就是给 AI 立规矩。规矩立好了，它干活才靠谱。**"

### Skills 和 MCP：按需加载的知识

"再往上是 **Skills** 和 **MCP**，"老王说，"这些我们之前都聊过了——Skills 是模块化的知识包，MCP 是连接外部世界的接口。在 Harness 体系里，它们的作用是'按需加载上下文'。"

"按需加载？"小明问。

"对！"老王说，"上下文是有限的，不能把所有知识都塞进去。所以 Harness 需要有一个机制——判断当前任务需要什么知识，然后把相关的 Skills 加载进来，把相关的 MCP 数据取进来。不需要的，就不加载，节省空间。"

### 上下文的生命周期管理

"最后，上下文管理还要管一件事——**上下文的生命周期**，"老王说。

"生命周期？"小明有点懵，"上下文还有生命周期？"

"当然有！"老王说，"上下文不是一成不变的。任务开始的时候加载什么，任务进行中补充什么，任务结束的时候保存什么，这些都要有明确的规则。"

比如：

- **启动时：**加载 System Prompt + 项目规则 + 任务描述
- **执行中：**按需加载 Skills、检索相关文档、补充工具返回结果
- **快满时：**压缩对话历史、总结已完成步骤、清理不需要的信息
- **结束时：**保存最终结果、记录关键决策、归档重要信息

上下文管理的本质，是给 AI 划定"信息边界"——  
它该知道什么，不该知道什么，什么时候知道，都得有规矩。

## 六、第四层：生命周期与编排 —— 任务的"项目经理"

"好了，前面三层都是'基础设施'——在哪跑、能用什么、能看什么，"老王喝了口水，继续说，"从第四层开始，就进入'任务管理'的范畴了。第四层——**生命周期与编排**。"

### 任务拆解：大任务拆成小任务

"除了生命周期管理，这一层还有一个重要职责——**任务拆解**，"老王说，"用户给的任务往往很大很模糊，比如'给我做一个登录页面'。这么大的任务，AI 没法一下干完。Harness 得帮它把大任务拆成小任务。"

"怎么拆？"小明问。

"有几种方式，"老王说，"有的是让 AI 自己拆——先让它制定一个计划，列出子任务清单。有的是预设好的工作流——固定的步骤，一步一步来。还有的是混合模式——大框架固定，细节让 AI 自己发挥。"

比如"做一个登录页面"这个任务，可以拆成：

1. 分析需求，确定登录页面的功能和样式
2. 创建登录组件的基础结构
3. 实现表单验证逻辑
4. 对接后端登录 API
5. 添加错误处理和加载状态
6. 编写单元测试
7. 运行测试，确保通过

小明

这就像……项目经理把大需求拆成一个个小任务，然后分配给开发人员？

老王

非常准确！Harness 的编排层，就是 Agent 世界的**项目经理**。它负责把大任务拆成小任务，安排执行顺序，跟踪每个任务的进度，确保最终能完成目标。

### 子代理管理：谁负责什么、怎么交接

"任务拆小了之后，如果还是干不完怎么办？"老王抛出一个问题。

小明想了想："那就……多找几个 Agent 一起干？"

"没错！"老王说，"这就是**子代理（Sub-Agent）**。一个大任务，可以拆给多个子代理并行处理。每个子代理负责一块，干完了把结果交回来，主代理再汇总。"

比如做一个完整的功能，可以拆成：

- **前端子代理**——负责写页面和交互逻辑
- **后端子代理**——负责写 API 和数据库操作
- **测试子代理**——负责写测试用例和跑测试
- **审查子代理**——负责代码审查和质量检查

"子代理管理听起来简单，其实学问很大，"老王说，"比如："

- 怎么分配任务？——每个子代理干什么活
- 怎么交接？——子代理干完了，怎么把结果传给下一个
- 怎么协调？——两个子代理需要同一份数据，谁先谁后
- 怎么纠错？——子代理做错了，谁来发现、谁来改

### 日志：AI 做了什么、说了什么

"最基础的是**日志**，"老王说，"AI 说的每一句话、调用的每一个工具、修改的每一个文件，都得记下来。"

"这有什么用？"小明问。

"太有用了，"老王说，"AI 做错了，你得知道它为啥做错吧？是理解错了需求？还是调用错了工具？还是工具返回的结果有问题？没有日志，你根本查不出来。"

日志通常记录这些内容：

- **对话日志：**AI 和用户的所有对话内容
- **工具调用日志：**调用了什么工具、参数是什么、返回了什么
- **文件操作日志：**读了哪些文件、写了哪些文件、改了什么内容
- **错误日志：**哪里出错了、错误信息是什么、堆栈信息

⚠️ 注意隐私

日志虽然重要，但也要注意保护隐私。敏感信息（比如密码、密钥、个人隐私数据）不能直接记在日志里，得脱敏处理。不然日志泄露了，比 AI 犯错还严重。

### 追踪：每一步花了多少时间、用了多少 token

"比日志更高级一点的是**追踪（Tracing）**，"老王说，"日志记录'发生了什么'，追踪记录'花了多少代价'。"

追踪主要关注：

- **时间：**每一步花了多长时间？总耗时多少？哪一步最慢？
- **Token：**输入多少 Token？输出多少 Token？总共花了多少钱？
- **调用链：**这个任务调用了哪些工具？先后顺序是什么？

"有了追踪数据，你就能做性能优化和成本优化，"老王说，"比如发现某个工具调用特别慢，你就可以优化它。发现某个任务 Token 消耗特别大，你就可以优化上下文。"

### 监控：成功率、错误率、成本

"再往上是**监控（Monitoring）**，"老王说，"追踪看的是单个任务，监控看的是整体趋势。"

监控的核心指标有：

✅

成功率

多少任务成功完成了？多少失败了？成功率是上升还是下降？

❌

错误率

最常见的错误是什么？哪类任务最容易出错？

💰

成本

每天花多少钱？平均每个任务多少钱？成本趋势如何？

⏱️

耗时

平均完成时间是多少？P95、P99 是多少？

老王

监控是给管理者看的。小明，你作为开发者可能更关心日志和追踪，但你的老板和项目经理，他们关心的是——这个东西靠谱吗？多少钱？效率怎么样？这些都得靠监控数据来回答。

### 审计：谁让 AI 做的、AI 做了什么、结果怎么样

"最高级的是**审计（Audit）**，"老王的语气变得严肃起来，"审计是给法务、合规、安全团队看的。"

审计回答三个问题：

1. **谁让 AI 做的？**——哪个用户、哪个角色、什么时候发起的任务
2. **AI 做了什么？**——调用了哪些工具、访问了哪些数据、修改了哪些文件
3. **结果怎么样？**——任务成功了还是失败了、有没有造成影响、影响有多大

可观测性的四层境界：  
日志告诉你"发生了什么"，  
追踪告诉你"花了多少代价"，  
监控告诉你"整体趋势怎么样"，  
审计告诉你"谁该为此负责"。

## 八、第六层：验证 —— "质检车间"

"第六层，**验证**，"老王说，"AI 干完活了，不能直接上线。你得检查检查——它做得对不对？质量过不过关？有没有引入新问题？"

### 自动化测试：跑完了对不对

"第一道验证关——**自动化测试**，"老王说，"这是最基础也是最重要的验证手段。AI 改完代码，你得跑一遍测试。测试全过了，才能进入下一关。测试没过，就得打回去改。"

单元测试

验证每个函数、每个组件是否正常工作。这是最基础的测试。

集成测试

验证多个模块配合起来是否正常。比如前端和后端能不能正常对接。

端到端测试

从用户的角度验证整个流程是否正常。比如从登录到下单的完整流程。

"自动化测试的好处是快、便宜、可重复，"老王说，"几秒钟就能跑完，发现问题马上就能反馈。但它也有局限——只能验证你写了测试用例的部分。没写测试用例的地方，它发现不了问题。"

### 静态检查：代码质量过不过关

"第二道验证关——**静态检查**，"老王继续说，"测试是'跑起来看看对不对'，静态检查是'不跑也能看出有没有问题'。"

静态检查包括：

- **代码规范检查（Lint）：**命名规不规范、格式对不对、有没有语法错误
- **类型检查：**TypeScript 类型对不对、有没有传错参数
- **安全扫描：**有没有安全漏洞、有没有敏感信息泄露
- **复杂度分析：**函数是不是太复杂了、有没有重复代码

💡 老王的建议

"静态检查一定要在 Harness 里自动跑。AI 写的代码，质量参差不齐。你让它自己检查自己，它肯定觉得自己写得都对。但工具是客观的——不符合规范就是不符合规范，有问题就是有问题。**用工具约束 AI，比用人约束靠谱多了。**"

### 独立审查：Reviewer Agent 来挑刺

"第三道验证关——**独立审查**，"老王说，"自动化测试和静态检查都过了，还不够。还得有一个'挑刺的'来专门找问题。"

"挑刺的？谁来挑？"小明问。

"另一个 Agent——Reviewer Agent（审查者代理），"老王说，"它不写代码，专门审查代码。让一个 AI 去检查另一个 AI 的工作。"

小明

这……靠谱吗？都是 AI，它能查得出来吗？

老王

哎，你别说，还真挺靠谱的。为什么？因为**写代码的 AI 和审查代码的 AI，视角不一样**。写代码的 AI 想着"怎么实现功能"，审查的 AI 想着"这里有没有问题"。视角不同，关注的点就不同。很多写代码时注意不到的问题，审查的时候一眼就能看出来。

Reviewer Agent 通常检查这些方面：

- **逻辑正确性：**实现逻辑对不对？有没有边界条件没考虑到？
- **代码质量：**写得好不好？有没有坏味道？能不能更简洁？
- **安全性：**有没有安全漏洞？会不会被注入攻击？
- **性能：**有没有性能问题？会不会很慢？
- **可维护性：**别人能看懂吗？好改吗？

### 人工审批：关键节点必须人来拍板

"最后一道验证关——**人工审批**，"老王说，"前面三关都是自动的，但有些事情，必须人来拍板。"

"哪些事情需要人来拍板？"小明问。

"凡是可能造成不可逆影响的，都需要人审批，"老王说，"比如："

- **部署到生产环境**——万一上线出问题，影响的是真实用户
- **删除数据**——数据删了就没了，恢复都恢复不了
- **修改核心逻辑**——核心逻辑改坏了，整个系统都可能崩
- **大额度操作**——比如发工资、转账，钱转出去就回不来了

验证的四层把关：  
自动化测试保功能，  
静态检查保质量，  
独立审查保严谨，  
人工审批保安全。  
一层比一层严格，一层比一层成本高。

小明

那我明白了……我上次那个事故，就是因为完全没有验证层。AI 说删就删了，连测试都没跑，更别说审查和审批了。

老王

对。有了验证层，就算 AI 犯了错，在上线之前也能被拦住。**验证层就是质量的"护城河"——越宽越深，越安全。**

## 九、第七层：治理 —— "交通管理局"

"终于到了最顶层——第七层，**治理**，"老王长长地舒了一口气，"如果说前面六层都是'技术层面'的管控，那治理就是'组织层面'的管控。"

▲ 治理体系：合规审计、权限治理、成本控制、持续优化——交通管理局的四大职能

### 合规审计：有没有违反规定

"治理的第一个职能——**合规审计**，"老王说，"公司有公司的规定，行业有行业的法规，国家有国家的法律。AI 干活，也得遵守这些规矩。"

📋

数据合规

AI 有没有违规使用用户数据？有没有泄露隐私？有没有违反数据保护法规（比如 GDPR、个人信息保护法）

🔐

安全合规

AI 有没有违反安全规定？有没有越权操作？有没有引入安全漏洞？

⚖️

伦理合规

AI 有没有生成有害内容？有没有歧视性输出？有没有违反伦理准则？

📊

审计跟踪

所有操作都要有记录、能追溯。出了问题，能查到是谁、在什么时候、做了什么。

老王

很多人觉得合规是"没事找事"，是"拖后腿"。但等你真的被罚了、被告了、数据泄露了，你就知道合规有多重要了。**合规不是限制你，是保护你。**

### 权限治理：谁能开什么级别的 Agent

"第二个职能——**权限治理**，"老王说，"不是所有人都能用最高级别的 Agent。不同的人、不同的角色、不同的部门，能使用的 Agent 级别不一样。"

比如：

- **普通员工：**只能用只读 Agent——查查资料、写写文档
- **开发人员：**可以用开发 Agent——写代码、跑测试、部署到测试环境
- **运维人员：**可以用运维 Agent——查日志、监控、但生产操作需要审批
- **管理员：**可以用全功能 Agent——但所有高风险操作都有审计记录

💡 最小权限原则

权限治理的核心还是"最小权限原则"——只给每个人完成工作必须的权限，多一点都不给。就像公司里，不是所有人都能进机房，不是所有人都能碰财务系统。Agent 的权限也一样。

### 成本控制：不能让 AI 把预算烧光

"第三个职能——**成本控制**，"老王说，"这个太重要了。AI 用起来很爽，但账单也很爽。你不控制成本，月底看到账单你会哭的。"

小明想起了自己那个月八千多的账单，心有余悸地点点头。

成本控制的手段包括：

- **预算限额：**每个团队、每个项目、每个用户，每月有预算上限。超了就停。
- **用量监控：**实时监控 Token 消耗和费用，快超了及时提醒。
- **成本优化：**分析哪些任务 Token 消耗大，能不能优化上下文、能不能用更便宜的模型。
- **分级计费：**简单任务用便宜的小模型，复杂任务才用贵的大模型。

没有治理的 Agent，就像没有交警的城市 ——  
看似自由，实则混乱。

### 持续优化：从失败中学习，改进 Harness

"最后一个职能——**持续优化**，"老王说，"Harness 不是搭好就完事了。它需要不断迭代、不断优化。"

"怎么优化？"小明问。

"从失败中学习，"老王说，"每次 AI 犯错，你都要问自己三个问题："

1. **为什么会犯这个错？**——是上下文不够？还是工具权限太松？还是验证没做好？
2. **哪一层没防住？**——这个错误本应该在哪一层被拦住？为什么没拦住？
3. **怎么改进 Harness？**——加什么规则、加什么检查、加什么验证，才能避免下次再犯？

真实案例

老王说，他认识的一个团队，每两周开一次"Agent 事故复盘会"。把这两周里 AI 犯的错、出的问题，全部拿出来复盘。然后针对性地改进 Harness——加规则、加检查、加测试。

一开始，每次复盘会都能找出十几个问题。半年之后，有时候一次会连一个问题都找不到。"不是 AI 变聪明了，而是 Harness 变完善了。**好的 Harness 不是一天建成的，是一次次从失败中打磨出来的。**"

## 十、七层联动，方得始终

讲到这里，老王放下笔，看着白板上密密麻麻的七层架构，对小明说：

老王

小明，七层都讲完了。你现在有什么感觉？

小明

感觉……好复杂啊！原来 Harness 有这么多层！我之前以为就是加个沙箱就完事了……

老王

是啊，Harness 是一个系统工程，不是一个单点功能。但你要记住——**七层不是孤立的，它们是层层联动、互相配合的。**

老王给小明举了个例子：

假设 AI 要删除一个文件。我们来看看七层是怎么配合的：

1. **第一层（执行环境）：**它只能在沙箱里删，影响不到外面
2. **第二层（工具接口）：**删除工具是红色工具，不能直接调用，必须确认
3. **第三层（上下文管理）：**System Prompt 里写了"删除文件前必须备份"
4. **第四层（生命周期）：**`before_write` 钩子触发，自动备份文件
5. **第五层（可观测性）：**记录删除操作的日志、时间、操作者
6. **第六层（验证）：**如果是重要文件，需要人工审批才能删
7. **第七层（治理）：**检查删除操作是否合规，有没有越权

好的 Harness 不是限制 AI，而是保护 AI ——  
保护它不犯致命的错，也保护我们不被它的错误伤到。

老王

你看，一个删除操作，七层都在发挥作用。每一层都有自己的职责，每一层都在为安全加一道锁。就算某一层失效了，还有其他层兜底。这就是"纵深防御"的思想。

小明

纵深防御……我懂了！就是不能把所有鸡蛋放在一个篮子里。不能只靠某一层来保证安全，得七层都做好，层层设防。

老王

没错！而且，**越往下的层，越基础，越重要**。沙箱没做好，上面六层再完善也白搭——AI 直接把系统搞崩了，你其他层再牛有啥用？所以做 Harness，一定要从下往上做，先把基础打牢。

小明看着白板上的七层架构图，久久不能平静。他想起自己那个"裸奔"的 Agent，想起那次删库事故，想起自己之前对 Harness 的轻视……

他掏出笔记本，认认真真地把七层架构抄了下来，然后在最后写了一行字：

✅ 本章金句

"Harness 的本质，是用确定性的系统，包裹非确定性的模型。"  
"没有治理的 Agent，就像没有交警的城市 —— 看似自由，实则混乱。"  
"好的 Harness 不是限制 AI，而是保护 AI —— 保护它不犯致命的错，也保护我们不被它的错误伤到。"

下章预告

### 一个人干活是安全了，但活太多干不完呢？

夕阳西下，会议室里只剩下小明和老王两个人。

小明合上笔记本，长出一口气。

"原来 Harness 有这么多层！有了这套系统，Agent 应该就能安全干活了吧？"

老王站起身，望着窗外的晚霞，嘴角露出一丝神秘的微笑。

"一个人干活是安全了。"老王慢悠悠地说，"但如果活太多，一个人干不完呢？"

小明愣住了。

对啊！一个 Agent 再安全、再能干，它也只有一双手、一个大脑啊。如果任务特别多、特别复杂，一个 Agent 根本忙不过来怎么办？

难道……可以让好多个 Agent 一起干活？

"那……那怎么让多个 Agent 一起干活啊？"小明急切地问。

老王拍了拍小明的肩膀，目光望向远方：
