一个人干不完，组队上（Sub-agents）

你有没有过这种感觉——一个 Agent 再厉害，好像也总有它搞不定的事？

小明最近就遇到了这个问题。

事情是这样的：公司要做一个新功能，叫"智能周报生成器"。小美给的需求是——员工什么都不用干，系统自动从代码提交、会议记录、任务看板里把每个人这周干了啥扒出来，整理成一份漂漂亮亮的周报，周五下班前自动发到群里。

小明一拍胸脯："这事儿简单，我让我的 Agent 去干！"

结果呢？三天过去了，Agent 还在那儿跟代码提交记录死磕呢。

不是 Agent 笨，是这活儿太杂了。要写前端页面，要写后端接口，要对接三个不同的数据源，要做样式美化，还要写测试用例。一个 Agent 既要当前端工程师，又要当后端工程师，还要当测试和产品经理——它能不懵吗？

小明挠着头去找老王："王哥，我发现一个 Agent 好像不够用啊。这功能要干的事儿太多了，它一会儿切到前端，一会儿切到后端，来回切换得特别慢，而且质量也不高。"

老王笑了："你这才发现啊？一个人干不完的活，那就——组队上。"

小明眼睛一亮："组队？你的意思是……让好几个 Agent 一起干？"

老王点点头："没错。这就是我们今天要聊的——**子代理（Sub-agents）**。"

## 10.1 为什么需要子代理？

### 一个 Agent 的能力是有限的

在讲子代理之前，我们先聊聊一个残酷的事实——**再厉害的 Agent，能力也是有限的。**

你可能会说："不对啊，大模型不是什么都懂吗？上知天文下知地理，前端后端都能写。"

话是这么说，但"懂"和"干得好"是两回事。

就好比你让一个全科医生去做心脏手术——他可能懂心脏的基本结构，但你敢让他开刀吗？你肯定更愿意找一个心外科专家，对吧？

Agent 也是一样的。一个通用型的 Agent 什么都能干一点，但什么都干不到极致。它有三个天生的局限：

🎯 三大局限

**上下文有限：**就像人的记忆力一样，Agent 的上下文窗口是有限的。你让它同时装下前端代码、后端代码、数据库结构、产品需求文档、测试用例……它装不下，装下了也记混了。

**精力有限：**Agent 一次只能专注做一件事。你让它写前端的时候，它就没法同时想后端的事。来回切换的成本很高，而且切换多了容易出错。

**专长有限：**虽然大模型什么都懂，但在某个具体领域，经过专门训练和配置的专家 Agent，肯定比通用 Agent 干得好。就像专业摄影师拍的照，就是比手机拍的好看。

小明点点头："有道理。我那个 Agent 就是这样——一会儿写前端，一会儿写后端，写着写着就把前端的变量名用到后端去了，bug 一堆。"

老王说："这就对了。不是它笨，是你给它的活太杂了。你让一个人同时干五个人的活，他能干好才怪。"

### 分工协作的力量

那怎么办呢？答案是——**分工。**

分工这件事，人类社会已经玩了几千年了。

原始社会的时候，大家都是什么都干——打猎、采果子、做衣服、盖房子，全能选手。但后来发现不对啊：有人打猎特别厉害，有人做衣服特别拿手，那不如让打猎厉害的专门打猎，做衣服好的专门做衣服，然后大家交换——这样整体效率高多了。

这就是分工的力量。**专家比通才更靠谱。**

好的子代理分工，是让每个 Agent 只做自己最擅长的事，然后通过清晰的交接串起来。

放到软件开发里，这个道理就更明显了。你看哪个正经公司是一个人全包的？都是前端团队、后端团队、测试团队、运维团队，各司其职。

为什么要分这么细？因为每一块都足够深，一个人不可能样样精通。前端要懂各种框架、性能优化、浏览器兼容；后端要懂数据库、架构设计、高并发；测试要懂各种测试方法、自动化工具。

Agent 也是一样的道理。你想让一个 Agent 既是前端专家又是后端专家还是测试专家——不是不可能，但是成本极高，效果还不一定好。

不如搞几个专门的 Agent：前端 Agent 专门写前端，后端 Agent 专门写后端，测试 Agent 专门测 bug。每个 Agent 只需要精通自己那一块，然后互相配合——这样整体效率和质量都上去了。

### 汽车比喻：一辆车坐不下所有人，组个车队

我们还是回到智能汽车的比喻。

前面我们说，一个 Agent 就像一辆会自己开的智能汽车。你告诉它目的地，它自己开过去。

但是——如果你的目的地是要搬一个家呢？

一辆小轿车再厉害，它能拉的东西也是有限的。沙发、冰箱、洗衣机、衣柜……你全塞进去？不可能，一辆车装不下。

那怎么办？**组个车队。**

一辆小轿车拉人，一辆小货车拉家具，一辆厢式货车大家电，再来个面包车拉零碎。各司其职，一起往目的地开。

## 10.2 子代理的"分工模式"

既然要分工，那怎么分呢？

这是个好问题。分工不是随便切的，切不好反而更乱。就像一个公司，如果部门划分不合理，大家天天扯皮，效率反而更低。

在子代理的世界里，常见的分工模式有三种。

▲ 三种经典分工模式：按阶段分、按领域分、按角色分

### 按阶段分：规划者 + 执行者 + 审查者

第一种分法，是按照**工作的阶段**来分。

任何一件复杂的事情，大致都可以分成三个阶段：先想清楚要怎么做，然后动手去做，最后检查做得对不对。

对应到子代理，就是三种角色：

- **规划者（Planner）：**负责制定计划。接到任务后，先分析一下这个任务要分几步，每一步干什么，先后顺序是什么。它不负责具体干活，只负责"想"。
- **执行者（Executor）：**负责具体干活。规划者说第一步干什么，执行者就去干什么。干完一步，回来汇报，然后等下一个指令。
- **审查者（Reviewer）：**负责检查质量。执行者干完活了，审查者来看看有没有问题、达没达标。有问题就打回去重做，没问题就通过。

你看，这是不是跟人类的工作模式一模一样？

老板接了个项目，先让项目经理做方案（规划者），方案定了让开发团队去做（执行者），做完了测试团队来测（审查者）。测完有 bug 打回开发改，改完再测，没问题了上线。

这种分工模式的好处是结构清晰、责任明确。每个 Agent 只需要关注自己那一段，不需要管全局。规划者不用关心具体怎么实现，执行者不用操心整体方向对不对，审查者只需要盯着质量。

🏭 流水线模式

按阶段分工，本质上就是一条流水线。任务从左边进去，经过规划、执行、审查三道工序，从右边出来成品。每一道工序只负责自己的环节。

### 按领域分：前端 Agent + 后端 Agent + 测试 Agent

第二种分法，是按照**专业领域**来分。

这个就更好理解了——你公司里前端组、后端组、测试组，不就是按领域分的吗？

在 Agent 团队里也是一样：

- **前端 Agent：**专门写前端代码。React、Vue、CSS、性能优化，它最懂。给它一个设计稿，它能给你写出漂漂亮亮的页面。
- **后端 Agent：**专门写后端代码。数据库设计、API 接口、服务架构、性能调优，它的地盘。
- **测试 Agent：**专门写测试用例、跑测试、找 bug。它的眼里只有"有没有问题"，天生自带找茬属性。

这种分工模式的好处是**专业度高**。每个 Agent 都深耕自己的领域，工具、知识库、prompt 都是为这个领域优化的。同样写一段代码，前端 Agent 写出来的质量，就是比通用 Agent 高。

小明插嘴："那我那个智能周报的项目，是不是就可以这么分？前端 Agent 写页面，后端 Agent 写接口，测试 Agent 测功能？"

老王点头："完全可以。而且这是最常见的分工方式——因为它跟我们人类团队的组织方式是一样的，大家容易理解。"

### 按角色分：产品 Agent + 架构 Agent + 开发 Agent + 运维 Agent

第三种分法，是按照**角色职能**来分。

你可能会问：这跟按领域分有啥区别？

区别在于——按领域分是"按技术栈分"，按角色分是"按职责分"。

举个例子，一个产品经理和一个架构师，他们可能都不写代码，但他们的角色完全不同。产品经理负责"做什么"，架构师负责"怎么做"。

在一个完整的 Agent 团队里，你可以配置这些角色：

🎯

产品 Agent

负责需求分析、功能设计、优先级排序，知道"做什么"

🏗️

架构 Agent

负责技术选型、系统设计、模块划分，知道"怎么做"

💻

开发 Agent

负责具体编码实现，把设计变成可运行的代码

🧪

测试 Agent

负责质量保证，找 bug、写测试、把关上线

🚀

运维 Agent

负责部署上线、监控告警、故障处理，保证稳定运行

你看，这是不是就是一个迷你版的互联网公司？产品、架构、开发、测试、运维，一条龙配齐。

这种分工模式适合做**完整的产品**，而不是单个功能。比如你说"帮我做一个 todo 应用"，从需求到上线全流程，这种角色分工就特别合适。

### 不是越多越好：分工太细反而增加协调成本

讲到这里，小明兴奋了："哇，那我搞十个八个 Agent，每个负责一块，是不是效率最高？"

老王泼了一盆冷水："别急。人多不一定力量大，Agent 多也不一定效率高。"

Agent 不是越多越好，就像团队不是人越多越好——人多了协调成本会吃掉效率。

为什么这么说？因为 Agent 之间要协作，协作是有成本的。

你想啊，如果只有两个 Agent，它们之间只需要一条沟通渠道。如果有五个 Agent 呢？那就是十条沟通渠道。十个 Agent 就是四十五条。渠道越多，信息传递越容易出问题，协调起来越费劲。

⚠️ 注意

每增加一个子代理，就会增加一份协调成本。任务交接、信息同步、冲突解决……这些都是要花时间和算力的。Agent 数量到了一定程度，增加 Agent 带来的收益，还不够抵消协调成本的增加。

这就是为什么"三个和尚没水喝"——不是人多不好，是**人多了组织不好反而更差**。

那多少个 Agent 最合适呢？

老王说："这个没有标准答案，看任务复杂度。简单的任务，一个 Agent 就够了。中等复杂度的，两三个。特别复杂的，五六个也就差不多了。再往上加，就要慎重考虑了。"

"记住一个原则——**能少用就少用**。能用一个 Agent 搞定的，就不要用两个。只有当一个 Agent 确实搞不定的时候，再考虑加人。"

小明点点头："有道理。就像我们做项目，不是人越多越快，有时候加人反而更慢。"

## 10.3 Builder + Reviewer：最经典的双人组

讲了这么多分法，老王话锋一转："但是，如果你是刚接触子代理，我建议你从最简单的开始——**两个人的组合**。"

"两个人？"小明问。

"对，两个人。一个干活，一个挑错。这就是最经典、最实用、也是性价比最高的子代理模式——**Builder + Reviewer**。"

▲ Builder + Reviewer：一个写代码，一个审代码——软件开发的黄金搭档

### Builder：负责干活，完成目标

先说说 Builder，也就是"建造者"。

Builder 的职责很简单——**把活干出来**。

你给它一个目标，比如"写一个用户登录页面"，它就吭哧吭哧去写了。写 HTML，写 CSS，写 JavaScript，写完给你交差。

Builder 的特点是：

- **行动力强：**说干就干，不墨迹。拿到任务就动手，速度快。
- **目标导向：**眼里只有"完成任务"，怎么快怎么来，怎么省事怎么来。
- **容易粗心：**因为光顾着往前冲，细节上容易出问题。边界情况没考虑到，代码写得有点糙，这些都是常有的事。

你看，像不像团队里那种"行动派"的工程师？交给他的活，干得快是快，就是有时候质量差点意思，得有人帮着把把关。

### Reviewer：负责挑错，默认怀疑

那谁来把关呢？就是 Reviewer——"审查者"。

Reviewer 的职责跟 Builder 正好相反——它不负责干活，它负责**挑毛病**。

Builder 把代码写出来了，扔给 Reviewer。Reviewer 拿过来，第一反应不是"写得好不好"，而是"有没有问题"。它戴着一副"找茬眼镜"，逐行检查：

- 这个逻辑有没有漏洞？
- 边界情况考虑到了吗？
- 变量命名规范吗？
- 有没有安全隐患？
- 性能会不会有问题？
- 测试覆盖到了吗？

Reviewer 的心态是——**默认怀疑**。你说你写好了？我不信，我得自己看看。找出问题算我的，找不出来算你的。

这种"怀疑一切"的态度，听起来挺讨厌的，但对质量保证来说太重要了。

不要让同一个 Agent 又当选手又当裁判——就像你不会让考生自己给自己打分。

### 为什么不能同一个人又当选手又当裁判

小明问："为什么非要分开呢？让 Builder 自己检查一下不行吗？"

老王摇摇头："不行。这里有个心理学效应，叫'**自己的刀削不了自己的把**'。"

"什么意思？"

"意思就是，自己写的东西，自己很难看出问题来。因为你写的时候，脑子里已经有了'这是对的'的预设。你再去检查，也是顺着自己的思路走，很难发现自己的逻辑漏洞。"

小明想了想："好像是这么回事。我写完代码自己测，总觉得没问题。但一交给测试，分分钟给我找出 bug 来。"

"就是这个道理。"老王说，"这不是能力问题，是**视角问题**。写代码的人和审代码的人，视角是完全不一样的。写的人想的是'怎么实现'，审的人想的是'哪里会出错'。这两种视角很难同时存在于一个脑子里。"

而且，除了视角问题，还有一个更根本的原因——**利益冲突**。

你想啊，如果让 Builder 自己审查自己的代码，它天然就有"赶紧通过"的动机。毕竟活是它干的，它当然希望越快越好、越少改动越好。你让它自己挑自己的毛病，它能尽心尽力吗？

这就像你考试完了自己给自己打分——你能做到完全客观吗？很难的，人之常情。

所以，必须把"干活的"和"挑错的"分开。Builder 负责往前冲，Reviewer 负责踩刹车。一推一拉，速度和质量就都有了。

💡 小知识

在软件工程里，这叫"职责分离"（Separation of Duties）。不只是代码审查，很多地方都用这个原则——比如财务管钱的和记账的不能是同一个人，就是怕监守自盗。

### 小明的实践：Bug 率降了一半

听老王讲完，小明决定回去试试。

他把自己那个智能周报项目重新调整了一下。不再让一个 Agent 从头到尾全包，而是搞了两个 Agent：一个叫 Builder-Frontend，专门写前端；一个叫 Reviewer-Frontend，专门审前端代码。

具体流程是这样的：

1. 主 Agent 把需求拆成一个一个小任务
2. 每个任务先交给 Builder 去做
3. Builder 做完了，把代码交给 Reviewer
4. Reviewer 检查代码，列出问题清单
5. 如果有问题，打回 Builder 修改
6. 改完了再交 Reviewer 复查
7. 直到 Reviewer 说"没问题"，这个任务才算完成

一开始小明还担心："这样来回打回，会不会很慢啊？"

结果跑了一周，小明惊了——

速度确实慢了一点，大概慢了 20%。但是——**Bug 率直接降了一半还多！**

以前一个 Agent 写的代码，小明自己测的时候，平均每个功能能找出 5、6 个 bug。现在 Builder + Reviewer 的组合，同样的功能，平均只有 2 个左右的 bug。而且剩下的 bug 大多是需求理解的问题，不是代码本身的问题。

小明跑去找老王："王哥，太神了！就加了一个 Reviewer，质量提升这么多！"

老王笑着说："正常。你想想，以前代码写完就完了，没人把关。现在多了一道质检关卡，质量能不上去吗？而且 Reviewer 是专门干这个的，它的 prompt 就是教它怎么找 bug、怎么查问题，专业度比 Builder 高多了。"

"那这 20% 的速度下降值不值呢？"小明问。

"你说呢？"老王反问，"以前你写完代码，是不是还要花大量时间自己测、自己改 bug？那些时间算进去了吗？"

小明一拍脑袋："对啊！我怎么忘了这个。以前虽然 Agent 写得快，但我自己改 bug 的时间老多了。现在虽然 Agent 那边慢了一点，但我这边省事多了。总体算下来，反而更快了！"

老王点点头："这就是 Builder + Reviewer 模式的妙处。看起来多了一道工序，实际上整体效率更高——因为**前期多花一点时间做质量控制，比后期出了问题再返工要划算得多**。"

## 10.4 子代理的协作机制

分工讲完了，接下来是个更重要的问题——**分了之后怎么协作？**

你想啊，一个团队不是人凑在一起就叫团队了。得有流程、有规矩、有沟通方式，不然就是一盘散沙。

子代理也是一样。你搞了五个 Agent，各干各的，谁也不理谁，那肯定乱套。必须有一套协作机制，让大家能配合起来。

▲ 子代理协作的四大机制：交接、共享上下文、通信协议、冲突解决

### Handoff：任务怎么交接

第一个问题：**任务怎么从一个 Agent 交到另一个 Agent 手里？**

别小看这个问题。交接交不好，后面全是坑。

比如你在公司里，A 同事做了一半的活交给 B 同事。如果 A 只是说"我做完了，你接吧"，那 B 肯定懵——做到哪了？哪些做了哪些没做？有什么坑要注意？下一步该干什么？

子代理之间也是一样。交接（Handoff）不是把球扔过去就行，得有**规范的交接清单**。

一个好的交接，至少要包含这些信息：

📋 交接清单

**任务背景：**我们在做什么事？目标是什么？

**已完成的工作：**我做了什么？做到什么程度了？

**产出物：**我生成了哪些文件/代码/文档？在哪？

**待办事项：**接下来要做什么？优先级？

**注意事项：**有什么坑？有什么风险？哪些地方要特别小心？

**问题与疑问：**有什么不确定的地方？需要对方帮忙决策的？

你看，这跟我们人类项目里的"交接文档"是不是一模一样？

老王说："交接是子代理协作里最容易出问题的环节。很多时候不是 Agent 能力不行，是**信息没传到位**。上一个 Agent 以为下一个 Agent 知道，下一个 Agent 以为上一个 Agent 已经做了——结果两边都没做。"

"所以，交接一定要**书面化、结构化、不留模糊地带**。宁可多说一句，不要少说半句。"

### 共享上下文：大家看的是同一份资料吗

第二个问题：**多个 Agent 怎么保证大家拿到的信息是一致的？**

这个问题很关键。你想啊，如果前端 Agent 看到的需求是 v1 版本，后端 Agent 看到的是 v2 版本，那写出来的东西能对得上吗？肯定对不上。

这就是"共享上下文"的问题——所有子代理都应该基于**同一份事实**来工作。

怎么做到呢？通常有几种方式：

- **单一信息源（Single Source of Truth）：**有一个"中心资料库"，所有 Agent 都从这里取信息。比如需求文档存在一个固定的地方，大家都读这份，谁也不用自己的版本。
- **变更广播：**如果有什么信息变了，要通知所有相关的 Agent。比如需求改了，不能只告诉前端，后端和测试也得知道。
- **上下文注入：**每次给子代理派任务的时候，把它需要知道的所有上下文都一起给它。别让它自己去找，找着找着就找错了。

小明插了一句："这不就是我们团队里的'信息同步'嘛。需求变更了要开同步会，文档要放在共享文档里，不能各存各的。"

老王点头："对。人类团队踩过的坑，Agent 团队一个都少不了。只不过人类是用会议和文档来同步，Agent 是用程序和协议来同步。本质是一样的。"

### 通信协议：子代理之间怎么说话

第三个问题：**Agent 之间怎么交流？**

你可能会说："它们不都是大模型吗？直接用自然语言聊天不就行了？"

理论上可以，但实际上——**纯自然语言交流效率太低，而且容易出歧义**。

就好比你在公司里，如果所有沟通都靠聊天，没有邮件、没有工单、没有文档，那肯定乱套。说过的话转头就忘，理解错了也没地方对质。

所以，子代理之间需要有一套**通信协议**——规定了谁跟谁说话、说什么格式、怎么确认收到、怎么回复。

常见的通信模式有几种：

| 通信模式 | 特点 | 适用场景 |
|-|-|-|
| 主从式 | 所有沟通都通过主 Agent 中转，子代理之间不直接说话 | 简单任务、少量子代理 |
| 点对点 | 子代理之间可以直接对话，灵活但容易乱 | 复杂协作、需要频繁沟通 |
| 黑板模式 | 大家都往一个"黑板"上写信息，谁都可以读 | 信息共享型任务、多方协作 |
| 消息队列 | 通过消息队列收发，异步解耦 | 大规模 Agent 团队、高并发 |

对于初学者来说，**主从式**是最稳妥的。所有子代理都只跟主 Agent 说话，子代理之间不直接联系。这样虽然效率不是最高的，但结构清晰、好管控、不容易乱。

💡 建议

刚开始玩子代理，就用主从模式。一个老大（主 Agent）管着几个小弟（子代理），老大派活，小弟干活然后汇报。简单直接，不容易出问题。等玩熟了，再考虑更复杂的通信模式。

### 冲突解决：意见不一致怎么办

第四个问题：**Agent 之间意见不一致，听谁的？**

小明笑了："Agent 也会吵架？"

老王也笑了："怎么不会？只要有两个以上的 Agent，就会有意见不一致的时候。比如 Builder 说'我这么写没问题'，Reviewer 说'不行，这里有 bug'——这不就是吵架吗？"

小明一想，还真是。

那意见不一致怎么办呢？总得有个裁决机制吧。

常见的冲突解决方式有这么几种：

- **主 Agent 裁决：**最简单的方式——谁官大听谁的。子代理吵起来了，主 Agent 来拍板。优点是快，缺点是主 Agent 不一定懂细节。
- **投票表决：**多个 Agent 投票，少数服从多数。适合有多个同角色 Agent 的情况，比如三个 Reviewer 一起审代码，两个说有问题那就有问题。
- **角色优先级：**不同角色有不同的话语权。比如在质量问题上，Reviewer 说不行就是不行，Builder 说了不算。在实现方式上，Builder 有更大的发言权。
- **人机协作：**实在拿不准的，就找人来决策。Agent 搞不定的事，交给人类来拍板。这是最稳妥的，当然也是最慢的。

老王说："实际应用中，通常是几种方式结合用。小问题角色内自己解决，中等问题主 Agent 裁决，重大问题或者拿不准的，就找人来。"

### 主 Agent 的角色：总指挥还是协调员

讲到主 Agent，小明问了一个问题："那主 Agent 到底是干什么的？它就是个发号施令的吗？"

老王摇摇头："没那么简单。主 Agent 的角色，取决于团队规模和任务类型。"

如果团队很小，比如两三个子代理，那主 Agent 更像是一个**总指挥**。它什么都管：任务怎么分、谁干什么、什么时候交接、出了问题怎么解决。它既是项目经理，又是技术负责人，还是产品经理。

但如果团队大了，比如有七八个、十几个子代理，那主 Agent 就管不过来了。这时候它得转型，从"总指挥"变成"**协调员**"。

什么是协调员？就是它不具体管每个子代理干什么，而是负责：

- 定大方向和目标
- 搭建团队结构和协作流程
- 解决子代理之间的冲突
- 协调资源和优先级
- 把控整体进度和质量

你看，这是不是就像一个公司的 CEO？CEO 不会去管每个员工具体干什么，他只负责定战略、搭班子、带队伍、控节奏。

"所以，"老王总结道，"主 Agent 的角色不是固定的。团队越小，主 Agent 越偏向'执行型领导'；团队越大，主 Agent 越偏向'管理型领导'。"

## 10.5 Worktrees：给每个 Agent 一个"工位"

聊完了协作机制，老王又抛出了一个新问题："小明，我问你——如果好几个 Agent 同时改同一个项目的代码，会发生什么？"

小明想都没想："那肯定冲突啊！两个 Agent 同时改同一个文件，改着改着就把对方的改动覆盖了。"

老王点头："对。所以子代理要一起干活，光有分工和协作还不够，还得有**工作空间的隔离**。"

"工作空间隔离？"

"对。每个 Agent 都得有自己的'工位'，不能大家挤在一张桌子上干活。"

▲ Worktrees：给每个 Agent 一个独立的"工位"，做完了再合并到主分支

### 什么是 Worktrees？—— 独立的工作目录

Worktree 这个词，最早来自 Git。Git 有个功能叫 `git worktree`，可以让你在同一个仓库里同时检出多个分支，每个分支有自己独立的工作目录。

听起来有点绕，大白话讲就是——**同一个项目，你可以同时开好几个"副本"，每个副本在不同的分支上，互相不干扰。**

比如你有一个项目，主分支是 main。你可以开一个 worktree 叫 feature-login，在上面做登录功能；同时再开一个 worktree 叫 feature-payment，在上面做支付功能。两个 worktree 是完全独立的目录，你改你的登录，我改我的支付，互不影响。

等各自做完了，再分别合并到主分支去。

这个概念借用到 Agent 领域，就是——**给每个子代理分配一个独立的工作目录**，让它在自己的目录里干活，不要直接在主目录上改。

### 为什么需要隔离：多个人改同一个文件会冲突

为什么非要隔离呢？直接让大家在同一个目录里干活不行吗？

还真不行。原因有三个：

⚠️ 不隔离的问题

**文件冲突：**两个 Agent 同时改同一个文件，后保存的会把先保存的覆盖掉。就像两个人同时编辑同一份文档，你改你的我改我的，最后只剩一个人的版本。

**状态干扰：**A Agent 写了一半的代码，B Agent 跑过来读，读到的是半成品，基于半成品继续开发，最后肯定出问题。

**责任不清：**出了问题不知道是谁改的。文件里混着好几个人的改动，出了 bug 找谁去？

你看，这些问题是不是特别熟悉？人类团队协作开发，也是一样的问题。所以我们才需要 Git，需要分支，需要 Pull Request。

Agent 团队也是一样的。而且 Agent 比人更需要隔离——因为 Agent 干活比人快多了，冲突起来也比人厉害得多。一个人改一个文件要几分钟，一个 Agent 几秒钟就改完了。两个 Agent 同时跑，分分钟给你造出十几个冲突。

### Git Worktree：用分支隔离工作区

那具体怎么实现隔离呢？最常用的就是 Git Worktree。

流程大概是这样的：

1. 主 Agent 接到任务，先在主仓库上切出一个新分支，叫 task/xxx
2. 给每个子 Agent 创建一个 worktree，对应一个子分支，比如 task/xxx-frontend、task/xxx-backend
3. 每个子 Agent 在自己的 worktree 里干活，改自己的文件
4. 子 Agent 干完了，提交自己的分支，发起一个 PR（Pull Request）
5. Reviewer Agent 审查代码，没问题就合并到任务分支
6. 所有子任务都完成了，主 Agent 把任务分支合并到主分支

你看，这跟我们人类用 Git 协作的流程几乎一模一样。

小明惊叹："这不就是把我们的开发流程照搬给 Agent 了吗？"

老王笑着说："对咯。软件工程发展了几十年，积累的最佳实践不是白来的。人类团队踩过的坑，Agent 团队一样会踩。所以直接把成熟的协作模式搬过来，是最稳妥的做法。"

💡 一句话

Worktree 不是什么黑科技，它就是 Agent 世界的"一人一个工位"。每个人在自己的工位上干活，做完了把成果交上去合并。简单、清晰、不容易乱。

### 比喻：每个人有自己的办公桌，做完了再合并到主分支

我们还是用办公室来打个比方。

想象一下一个办公室，里面有好几个人一起做一个项目。

如果没有工位的概念——大家共用一张桌子，共用一份文档。你写两行，我写三行，他删一段。那最后文档成什么样？根本没法看。

但如果每个人都有自己的办公桌呢？

- 每个人在自己的工位上干活，用自己的草稿纸
- 写完一部分，觉得没问题了，就把这部分内容抄到主文档上去
- 抄之前还要让同事帮忙看看有没有问题
- 没问题了，才正式合并到主文档

这样是不是就井井有条了？

Worktree 就是每个 Agent 的"办公桌"。主分支就是"主文档"。Agent 在自己的办公桌上干活，干得差不多了，经过审查，再合并到主文档里去。

小明点点头："明白了。原来 Worktree 不是什么复杂的东西，就是给每个 Agent 一个独立的工作空间，避免大家互相干扰。"

老王竖起大拇指："好问题。这就涉及到 Agent 团队的两种形态——**临时子代理**和**固定团队**。"

▲ Agent Teams：从临时组队到固定编制，Agent 团队也在走向"正规化"

### 临时子代理 vs 固定团队

先说说临时子代理。

什么叫临时子代理？就是——**有任务了才创建，任务做完就销毁**。

就像工地里的临时工。有活了，招一批人来干活。活干完了，大家拿钱走人，各奔东西。下次再有活，再重新招人。

这种模式的好处是灵活。任务大就多招几个，任务小就少招几个。不需要的时候一个都不用养，省钱。

但坏处也很明显——**每次都要重新磨合**。新人不知道规矩，不知道项目背景，不知道怎么配合。前几天效率肯定低，还容易出问题。

那固定团队呢？

固定团队就是——**团队结构是预先定义好的，谁负责什么、汇报给谁，都有明确的规定**。不管有没有任务，团队编制都在那儿。

就像公司里的正式团队。产品组、前端组、后端组、测试组，编制固定。来活了，大家一起上；没活了，大家各自学习、优化、做技术储备。

这种模式的好处是**稳定、高效、配合默契**。大家在一起久了，知道彼此的习惯，知道流程怎么走，配合起来行云流水。

坏处呢？就是成本高。哪怕没活干，团队也得"养着"——当然 Agent 的"养着"不像人那么费钱，但配置、维护、管理也是有成本的。

| 对比维度 | 临时子代理 | 固定团队 |
|-|-|-|
| 灵活性 | 高，按需组建 | 低，编制固定 |
| 效率 | 低，需要磨合 | 高，配合默契 |
| 成本 | 低，用完即走 | 中，需要维护 |
| 质量稳定性 | 不稳定，看运气 | 稳定，有保障 |
| 适用场景 | 一次性任务、探索性工作 | 重复性工作、长期项目 |

### 团队配置：谁负责什么、汇报给谁

如果是固定团队，那就要好好设计一下团队结构了。

一个典型的软件开发 Agent 团队，大概长这样：

🏛️ 主 Agent（项目经理）

🎯 产品 Agent

🏗️ 架构 Agent

🧪 测试 Lead

🚀 运维 Agent

💻 前端 Agent A

💻 前端 Agent B

⚙️ 后端 Agent A

⚙️ 后端 Agent B

🧪 测试 Agent

你看，跟人类团队的组织结构图是不是很像？

每个角色都有明确的职责和汇报关系：

- 主 Agent 总管全局，所有人向它汇报
- 产品 Agent 负责需求和功能设计
- 架构 Agent 负责技术方案和系统设计
- 前端组、后端组、测试组各有分工
- 运维 Agent 负责部署和线上稳定性

老王说："团队配置没有标准答案，完全看你的需求。做网站的团队和做数据的团队，配置肯定不一样。关键是——**职责要清晰，汇报关系要明确**。不能谁都管谁，也不能谁都不管谁。"

### 并行工作：多个子代理同时干活

固定团队的一大好处是——可以**并行工作**。

什么意思呢？就是多个子代理可以同时干不同的事，不用等一个干完了另一个再上。

比如做一个新功能：

- 产品 Agent 在写需求文档的时候，架构 Agent 就可以开始做技术预研了
- 前端 Agent 在写页面的时候，后端 Agent 可以同时写接口
- 开发在写代码的时候，测试 Agent 就可以开始写测试用例了

这样一来，整个项目的周期就大大缩短了。如果是串行的，可能要 10 天；并行的话，可能 5 天就搞定了。

小明眼睛亮了："这就是'人多力量大'的正确打开方式啊！"

老王笑着说："对。但并行不是想当然的，得有条件。"

什么条件呢？

- **任务可拆分：**任务得能拆成几个独立的部分，各干各的不打架。如果是强依赖的，比如必须先有后端接口前端才能做，那就没法完全并行。
- **接口要先行：**虽然可以并行做，但大家得先约定好"接口"。前端和后端得先定好 API 长什么样，然后各自去实现，不然到时候对不上。
- **同步要及时：**并行不代表各干各的。定期同步进度、同步变更，不然到最后发现大家做的不是一回事，那就惨了。

### 怎么避免"三个和尚没水喝"

讲到并行，小明又想到了一个问题："王哥，那人多了，会不会出现'三个和尚没水喝'的情况？大家都觉得别人会去做，结果谁都没做。"

老王点头："这个问题问得好。人多了，确实容易出现责任扩散的现象。Agent 多了也一样。"

那怎么避免呢？关键是三点：

✅ 三个原则

**1. 任务到人，责任明确：**每件事都要有明确的负责人。不能说"大家看着办"，要说"这件事 A 负责，B 协助"。出了问题找谁，清清楚楚。

**2. 进度透明，定期同步：**每个人在干什么、干到哪了，所有人都能看到。有进度看板，有每日站会，谁在摸鱼一目了然。

**3. 结果导向，奖惩分明：**干得好的有奖励，干得差的有惩罚。Agent 虽然没有情绪，但可以通过权重、优先级来体现。表现好的 Agent 下次多派活，表现差的减少任务甚至淘汰。

"记住，"老王说，"**好的团队不是靠自觉，是靠机制**。人类团队如此，Agent 团队也如此。"

## 10.7 Agent Swarms：蜂群思维是真的吗？

聊到这里，小明突然想起了什么："对了王哥，我最近在网上看到一个词叫'Agent Swarm'，说什么几百个 Agent 一起干活，蜂群思维什么的。这是真的吗？"

老王笑了："你说的这个，是子代理的终极形态——**Agent Swarm（智能体集群）**。"

▲ Agent Swarm：成百上千个 Agent 协同工作——是未来可期，还是噱头大于实质？

### 什么是 Agent Swarm：几十上百个 Agent 一起干活

什么是 Swarm？直译过来就是"蜂群"或者"蚁群"。

你观察过蚂蚁吗？一只蚂蚁很笨，但是一群蚂蚁在一起，就能做出很复杂的事情——找到食物最短路径、建造复杂的巢穴、组织大规模迁徙。没有谁指挥，大家都是跟着简单的规则走，但整体行为却呈现出惊人的"智能"。

这就是"蜂群思维"（Swarm Intelligence）——**大量简单个体通过简单规则交互，涌现出复杂的群体智能**。

Agent Swarm 就是借鉴了这个思路。不是一个老大管一群小弟，而是一大堆 Agent，每个都遵循简单的规则，自己找活干、自己协作，最后完成复杂的任务。

听起来是不是很科幻？

更科幻的是，这已经不是概念了——已经有人在做了，而且还做出了一些让人吃惊的成果。

### Cursor 造浏览器、Anthropic 造 C 编译器

老王给小明举了两个例子。

第一个是 Cursor。你可能知道 Cursor 是个 AI 编辑器，但你知道吗？Cursor 团队做过一个实验——**用大量 Agent 协作来构建一个浏览器**。

对，你没听错，造浏览器。不是什么玩具浏览器，是真的能跑网页的那种。

怎么干的呢？他们搞了一堆 Agent，有的写 HTML 解析器，有的写 CSS 渲染引擎，有的写 JavaScript 引擎，有的写网络层……几十个 Agent 同时干活，最后真的拼出了一个能工作的浏览器。

小明张大了嘴："这也太夸张了吧？造浏览器可是大工程啊！"

"是挺夸张的。"老王说，"不过你别急着惊叹，我再给你说第二个。"

第二个是 Anthropic 做的实验——**用 AI Agent 从零开始写一个 C 语言编译器**。

C 编译器是什么概念？这可是计算机科学里最硬核的东西之一。几十年来，无数顶尖工程师前赴后继才把 GCC、Clang 做到今天这个水平。

而 Anthropic 用 Agent Swarm 的方式，让一堆 Agent 分工协作，从零开始写出了一个能工作的 C 编译器。虽然性能和成熟度比不上 GCC，但功能是完整的——能把 C 代码编译成可执行程序。

小明已经说不出话了。

### 为什么这些任务能成：因为有明确的 spec 和测试

过了好一会儿，小明才回过神来："王哥，这也太牛了吧？那是不是以后什么都能让 Agent Swarm 干了？"

老王摇摇头："别急着下结论。你有没有想过——为什么造浏览器、写编译器这些任务能成？"

小明想了想："因为……这些任务比较明确？"

"对！"老王一拍桌子，"关键就在这儿。这些任务有两个共同点：**有明确的规范（spec），和有明确的测试（test）**。"

什么意思呢？

比如 C 编译器，它的规范是非常明确的——C 语言标准就摆在那儿，什么语法是什么意思，清清楚楚。而且测试也是明确的——一段代码输进去，编译出来的程序运行结果对不对，一跑就知道。

浏览器也是一样。HTML、CSS、JavaScript 都有 W3C 标准，清清楚楚。而且有大量的测试用例，可以验证浏览器的每个功能对不对。

在这种"规范明确、测试明确"的场景下，Agent Swarm 特别好用。因为：

- 每个 Agent 都知道自己要做什么（规范写死了）
- 做得对不对一测就知道（测试写死了）
- 出了问题能快速定位（哪个测试没过就是哪块的问题）
- 可以大量并行（每个 Agent 负责实现一条规范）

"说白了，"老王总结，"这些任务就像搭乐高——每一块长什么样、跟谁拼接，都是规定好的。你有 100 个人，每人拼一块，最后拼起来就是完整的。人越多越快。"

### 企业级应用还早：别盲目跟风

"但是——"老王话锋一转，"如果你以为 Agent Swarm 已经成熟了，可以直接拿来做业务了，那你就想多了。"

为什么？因为**现实世界的大部分问题，都没有那么明确的规范和测试**。

比如你要做一个"用户喜欢的产品"——什么叫"喜欢"？怎么定义？怎么测试？说不清楚。

比如你要做一个"好的营销策略"——什么叫"好"？转化率高就是好？品牌提升就是好？标准是什么？

比如你要写一篇"爆款文章"——什么叫"爆款"？10万+ 就是爆款？内容质量怎么衡量？

这些问题，没有标准答案，没有明确规范，也没有自动化测试。你说做对了？可能换个人就说你做错了。

在这种场景下，Agent Swarm 就很难发挥作用了。因为大家连"目标是什么"都没达成一致，怎么分工？怎么协作？怎么验证对错？

⚠️ 现实很骨感

目前 Agent Swarm 的成功案例，大多集中在"规范明确、可自动测试"的领域，比如编译器、浏览器、数学证明、游戏 AI 等。而企业里的大部分业务问题，都是模糊的、开放的、没有标准答案的——这些场景，Swarm 还很难落地。

而且还有一个更现实的问题——**成本**。

一个 Agent 跑一次可能才几毛钱。但如果是 100 个 Agent 呢？跑一次就是几十块。如果是 1000 个呢？几百块。而且 Swarm 不是跑一次就完了，它要不断迭代、不断试错，那成本就嗖嗖往上涨。

对于企业来说，这成本能不能覆盖收益？还真不好说。

所以老王的建议是：

Agent Swarm 是未来，但未来还没来。与其追蜂群的噱头，不如先把几个子代理的协作玩明白。把基础打好了，等真的需要 swarm 的时候，自然水到渠成。

小明点点头："明白了。就像学武功，先把扎马步练好了，再去想什么神功盖世。"

老王哈哈大笑："你这比喻，绝了。"

## 本章小结

这一章我们聊了子代理——也就是"让多个 Agent 组队干活"这件事。

我们从"为什么需要子代理"讲起，说了单个 Agent 的三大局限：上下文有限、精力有限、专长有限。分工协作是突破这些局限的最好办法。

然后我们聊了三种分工模式：按阶段分（规划者+执行者+审查者）、按领域分（前端+后端+测试）、按角色分（产品+架构+开发+测试+运维）。同时也强调了——Agent 不是越多越好，分工太细反而增加协调成本。

接着我们深入讲了最经典的双人组——Builder + Reviewer。一个负责干活，一个负责挑错。为什么要分离？因为自己的刀削不了自己的把，又当选手又当裁判肯定出问题。

然后我们聊了子代理的协作机制：任务交接要清晰、上下文要共享、通信要有协议、冲突要有裁决。主 Agent 的角色从"总指挥"到"协调员"，取决于团队规模。

我们还聊了 Worktrees——给每个 Agent 一个独立的工作目录，避免大家改同一个文件造成冲突。就像办公室里每个人有自己的工位一样。

最后我们聊了 Agent Teams 和 Agent Swarms。从临时组队到固定团队，从几个 Agent 到成百上千个 Agent。也泼了冷水——Swarm 虽然酷炫，但企业级应用还早，别盲目跟风。

📝 本章金句

1. "不要让同一个 Agent 又当选手又当裁判——就像你不会让考生自己给自己打分。"

2. "Agent 不是越多越好，就像团队不是人越多越好——人多了协调成本会吃掉效率。"

3. "好的子代理分工，是让每个 Agent 只做自己最擅长的事，然后通过清晰的交接串起来。"
